# Audio-to-WAV is RECORD-based (a real-time run capture), not an offline export

picopilot gets a WAV out of a cart by RECORDING a running, cooperating cart (`extcmd("audio_rec")` / `extcmd("audio_end")`), NOT by PICO-8's offline headless WAV export. The offline path (`pico8 <cart> -export foo.wav`) is BROKEN upstream (a cross-platform SIGFPE crash, no file), so audio-to-WAV ships as two record-based surfaces: **`picopilot audio record <cart>`** (record the whole run's audio) and **`picopilot run --record-audio`** (capture audio alongside a normal run); and **`picopilot audio render`** (US #13, "hear my composed sfx/pattern") ships as a THIN layer over the record primitive that injects a play-harness for the selected target. Verified by a spike (`work/notes/findings/pico8-audio-wav-export-and-record.md`).

## Considered Options

- **Offline `-export` WAV (the intended US #13 path, REJECTED, broken upstream).** `pico8 <cart> -export foo.wav` reproducibly SIGFPEs (exit 136) and writes nothing, on both `pico8` and `pico8_dyn`, with a display (SIGFPE) or without (fatal SDL). Ruled out as OUR problem: not static-vs-dynamic linking, not a missing dependency (`ldd` clean), not display/config. Confirmed a KNOWN, cross-platform PICO-8 limitation (BBS tid=50123: same crash on Windows, filed as an unfulfilled feature request), and even absent the crash, `EXPORT FOO.WAV` depends on interactive editor state that does not exist headless, so the render-target selection has no verifiable offline mapping. There is simply no working offline `cart → WAV` in PICO-8 v0.2.7.
- **Record a running cart via `audio_rec`/`audio_end` (CHOSEN).** VERIFIED to produce a real WAV (5.34 s / 117,760 frames / 22050 Hz mono captured from a `-run` session). This is the audio sibling of how `picopilot run` captures screenshots from a cooperating cart, so it reuses that orchestration shape (launch + sentinel-watch + collect an artifact).
- **`audio render` as its OWN offline codec (REJECTED, nothing to build on).** With no offline export, there is no offline WAV to synthesize without reimplementing PICO-8's entire synth/mixer in TS, a perpetually-stale effort explicitly out of scope (mirrors the shrinko8 shell-out reasoning, ADR-0001). Emulating render over the record primitive delivers the US #13 value now without that.

## Consequences

- **Recording needs a REAL (or virtual) audio+video session; it does NOT work under headless `-x`.** VERIFIED: under `pico8 -x`, `audio_rec`/`audio_end` produce a well-formed but EMPTY (0-frame) WAV, headless PICO-8 mixes no audio to capture (same with `SDL_AUDIODRIVER=dummy`, longer windows, sfx or music); only a live `-run` session captures real audio. So this feature targets a developer's real machine (option i): the live capturing run is a MANUAL / opt-in test tier, exactly as `picopilot run` treats live PICO-8. CI covers only the orchestration against a fake runner + the structured `pico8-not-found` failure. A virtual-A/V-session harness (Xvfb + a PulseAudio null-sink) that would make recording headless/CI-automatable (option ii) is captured as `work/notes/ideas/audio-record-virtual-av-session.md`, NOT built now.
- **`audio render` (US #13) is EMULATED on the record primitive, not deferred.** It injects a throwaway harness cart (same `__sfx__`/`__music__`) whose `_update` plays the selected target (`sfx N` / `music P` / whole song), records around it, and returns the WAV, so the author writes no playback code and still gets "render sfx 5 → a WAV". It shares the record machinery; render and record stay DISTINCT in intent (render injects a play-harness for a chosen target; record captures a cart's own playback).
- **HONEST framing is required in the command help.** `audio render` is a REAL-TIME recording of playback (needs a real audio session), NOT a deterministic offline export. This must be stated so an author is not misled by the word "render" into expecting offline determinism. The `picopilot-audio` skill carries the same caveat.
- **`audio_end(1)` saves to PICO-8's current folder, NOT the `-desktop <dir>` path** (unlike screenshots/gifs). The command controls the WAV location via PICO-8's working folder / `root_path` (or `audio_end(0)` + `-desktop`), and isolates it to a temp/scratch dir in tests (the shared-write discipline).
- **Swap-in path preserved.** If Lexaloffle ever fixes offline WAV `-export`, an offline exporter can slot in UNDER the `audio render` surface without changing its command shape, the record-based implementation is an interim, not a commitment to real-time-only forever.
- This partially SUPERSEDES the spec's US #13 framing ("export a WAV"): the capability ships, but as a record-of-a-run rather than an offline export, reshaped by the spike exactly as ADR-0006 reshaped `run` and ADR-0005 anticipated ("reshape before committing the command surface").
