# `picopilot export` is a THIN structured wrapper over `pico8 -export`, standalone-first with a payload-only split

`picopilot export <cart> [dest]` shells out to PICO-8's own HTML export over the HEADLESS path (`pico8 <cart> -export <dest>/index.html -x`, no display needed, verified running with `DISPLAY` unset) and collects the produced pair into one structured result: the shell `index.html`, the Emscripten runtime + baked-in cart payload `index.js`, the file list, and a `labelWarning` flag. It names the pair `index.html` + `index.js` so a `dest` folder (e.g. `./website/static/games/<slug>/`) is directly serveable and showcase-ready. PICO-8 owns the bundle; picopilot only adds the glue (resolve cart, pick/create the out dir, name the pair, structure the result, surface the labelless-cart warning), mirroring the `run` stance (ADR-0006). It hard-requires the paid binary; absence is the first-class `pico8-not-found` value + a nonzero exit (never a crash), the CI-testable boundary via the injected adapter/spawn seams.

## Considered Options

- **No command, skill-only (rejected, partially true).** Raw `pico8 -export` IS PICO-8's own CLI and adds no value if merely echoed; that belongs in a skill. What tips it to a command is the orchestration an agent should not hand-roll: headless invocation, `index.html`/`index.js` naming for a directly-serveable folder, file collection, the labelless-cart warning, and the same `pico8-not-found` structured boundary as `run`.
- **Emit pico8's default `<cartname>.html` naming (rejected).** Naming the pair after the cart forces the website (or a human) to rewrite references and rename per game. Passing `index.html` to `-export` yields a folder that plays as-is and drops cleanly under `static/games/<slug>/`, with the shell page already referencing its sibling `index.js` by name (verified against real export output).

## Consequences

- **A `__label__` is required for a good bundle.** A labelless cart prints "please capture a label first" and produces a blank/ugly splash (verified); worse, some cases write nothing. So `export` surfaces `labelWarning` as a soft CTA and treats a missing `.js` payload as a hard `export-failed` error rather than a hollow success.
- **`--label <128x128.png>` bakes a splash in without an interactive F7.** Because a labelless cart is exactly the manual PICO-8 chore picopilot should absorb, `export --label <png>` decodes a 128x128 PNG to nearest-palette indices (a dependency-free PNG decoder + the existing palette, `engine/gfx/label.ts`) and injects it as `__label__` into a THROWAWAY copy written NEXT TO the cart (so `#include` still resolves), exports that, and cleans the copy up. The user's cart is never mutated. Label precedence: an existing `__label__` WINS (the flag is a fallback, not an override); else `--label`; else the fail-loud CTA. The screenshot-a-frame variant (`--label-frame`, auto-capture from the running game) is a queued idea (`work/notes/ideas/export-label-frame-auto-screenshot.md`), heavier because it needs the live run + a playtest-style harness transform. The default when a labelless cart is exported with NO label flag stays fail-loud (a game deserves a real splash), just with a one-flag remedy.
- **Two shapes, one command.** Default is STANDALONE (`index.html` + `index.js`, directly playable, what `serve` uses internally). `--payload-only` keeps only the `.js` runtime payload (a post-export file op the command owns, verified: the html is a disposable shell that only references the sibling `.js`, and the `.js` hardcodes no name) for a website that provides its OWN player shell.
- **The website, not the CLI, owns the player component.** picopilot's job stops at producing the payload/bundle; a Svelte `Pico8Player` component (in `./website`, NOT the picopilot engine) ports PICO-8's small inline bootstrap (a `Module` config + canvas) and loads an exported `.js`. `--payload-only` exists precisely to feed that component. The CLI reimplements nothing of PICO-8's runtime.
- **Showcase population is manual.** There is no CI export step (PICO-8 is a paid binary). You point `export <cart> ./website/static/games/<slug>/` at the games you want to showcase; the site build just serves what is present.
- Same second run/export architecture note as ADR-0006 applies: a headless-browser path over this export is not foreclosed; this command targets the native headless export first.
