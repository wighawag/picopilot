# `picopilot playtest` drives arbitrary carts by transforming their input to a serial channel, over a fixed-block + ACK transport

`picopilot playtest` drives an ARBITRARY PICO-8 cart through scripted input and captures live gameplay (not the title screen) by transforming a THROWAWAY copy of the cart: it redefines the cart's global `btn`/`btnp` to read a per-frame held-buttons byte from `serial(0x804)` (reconstructing `btnp` edges from the level signal), and wraps `_update`/`_update60`/`_draw` so the harness owns the frame loop (advance exactly N frames, then freeze). The host<->cart transport is the load-bearing decision recorded here: FIXED-SIZE command blocks over stdin + a one-command-per-frame budget-gated drain + a stdout ACK handshake. All of this was spike-verified on PICO-8 v0.2.7 (headless `-x`).

## Context

An agent building a game with picopilot could boot a cart (`run`) but not DRIVE it through play: `run` screenshots the title, and `run --input`'s one-shot `stat(6)` only works if the cart cooperates (an arbitrary game reads real `btn`/`btnp`). The game-jam benchmark made this concrete: the judge could only ever see the title screen. The mechanism to fix it (transform the cart's input to a host-driven channel) was prototyped in the bench, then the design was grilled and spiked to resolution.

The naive "keep pico8 alive and pipe input to it" was FLAKY: small unpadded writes to a live `pico8 -x` stdin coalesce/drop in the OS pipe buffer (verified: 1-2 byte writes were lost/merged; a 16-byte block per flush delivered reliably every time). `printh`/stdout, by contrast, streams live and reliably (cart->host). That asymmetry shaped the transport.

## Decisions

- **Input by transform, not injection.** Redefine the cart's global `btn`/`btnp` (they are just globals; shadowing overrides the built-ins, verified) to read `serial(0x804)`. RECONSTRUCT `btnp` EDGES from the held level (track the previous frame; emit true only on 0->1), so a single scripted press triggers exactly one `btnp` (menus/start/single-actions behave), not a held-repeat. This makes any normally-written game drivable with NO change to its own cart.
- **The harness owns the frame loop (explicit frame-stepping).** Wrap `_update`/`_update60`/`_draw`; advance exactly N frames on a STEP, and when the budget is 0 the game FREEZES (its callbacks skipped -> all state held, incl. logic wrongly placed in `_draw` -> a stable framebuffer for a clean screenshot). No-op-safe when a callback is missing (draw-only carts). Verified on normal, logic-in-draw, and draw-only carts. This makes captures DETERMINISTIC rather than timing-dependent (the prototype's title-vs-gameplay flakiness), and it makes one-shot and the resumable session the SAME cart-side machine.
- **Transport = FIXED-SIZE command blocks + one-per-frame drain + a stdout ACK handshake.** (1) Send fixed-size blocks (opcode + args + pad), because small writes coalesce/drop in the pipe. (2) The cart reads exactly one block per frame and drains it against the step-budget, so commands queue in order and none are lost. (3) The cart `printh`s an ACK when a command completes; the host WAITS for the ACK before sending the next command. This is fully deterministic with no wall-clock guessing. Opcodes: STEP<n> / INPUT<byte> / SHOT / PAUSE / QUIT; the shim buffers partial blocks across frames.
- **Determinism for stateless replay is opt-in via `--seed`.** `srand(k)` fully determinizes `rnd()` (verified: identical sequences across runs; no `srand` auto-randomizes the seed on startup). The transform can inject `srand(n)` at cart start (opt-in `--seed`, never silent because it changes game behaviour), so an otherwise-random cart is replay-safe. It stays deterministic even for carts that reseed from DETERMINISTIC state (`srand(t)` replays identically under frame-stepping); it only breaks for the rare cart that reseeds from a NON-deterministic source (wall-clock/entropy) mid-run.

## Considered Options

- **Inject real button presses (poke `0x5f4c`) instead of transforming (rejected).** Poking the button-state memory DOES make `btn()` read pressed, but reliably producing `btnp` EDGES that advance a specific game from title into play is game-specific and did not generalize; the input-transform is general and deterministic.
- **`stat(6)`/`-p` one-shot or the `-i` file channel (rejected for driving).** `stat(6)` needs cart cooperation; the `-i` (`serial 0x806`) channel delivered data but buffered file-like (all-at-once), not live per-frame. Live stdin `0x804` with fixed blocks is the working live channel.
- **A wall-clock-timed input stream (rejected).** Spacing input by real time is exactly what made the prototype flaky (commands coalesced, captures landed on the wrong frame). The block + ACK handshake replaces timing with an explicit request/response.

## Consequences

- A NEW `picopilot playtest` command (not a flag on `run`; `run` stays the thin boot-and-capture command). It is PICO-8-gated (structured `pico8-not-found` + nonzero when absent, mirroring `run`/`audio render`); the CI-testable surface is the pure transform + the btnp-edge + the block codec + `--seed` determinism, and the live drive-and-capture is a manual/opt-in tier.
- Two resumability paths fall out of the same machine: ONE-SHOT (send the whole opcode script up front, exit) and a LIVE resumable SESSION (keep the driven process alive + paused between the agent's turns, addressed by a session id, over the same block+ACK transport). A third, STATELESS REPLAY, is FREE (re-invoke one-shot with an accumulating script + `--seed`), so it is documented rather than built.
- Transform-safety is best-effort with a documented caveat: a cart that reassigns `_update` at runtime or drives its own coroutine loop may not pause/step correctly. `playtest` is a jam/debug tool, not a sandbox.
- The game-jam benchmark's bespoke `drive-capture.sh` is superseded by the command (the drive logic now lives in one tested place).
