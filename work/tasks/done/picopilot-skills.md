## What to build

The agent-discovery surface. Ship the `picopilot-*` skills so an agent gets the hard-won workflow knowledge on-demand by command group (incur's `sync: { depth: 1 }` — one skill file per top-level group): `picopilot-overview`, `picopilot-code`, `picopilot-art`, `picopilot-audio`, `picopilot-debug`. Each carries its group's discipline: the `#include` discipline, reading the token breakdown, the `gfx render`→look→`gfx set`→fix loop INCLUDING the non-multimodal fallback ("view the PNG if you can read images, else reason over the `gfx show` grid"), composing in picopilot-MML.

Then wire `picopilot init --install-skills` — the explicit opt-in that runs incur's `skills add` to actually install the skills into the agent's skill dirs. This is the ONLY path in the tool that writes to a SHARED/GLOBAL location (incur's `skills add` defaults to a GLOBAL install), so it carries the shared-write isolation obligation. Default `init` (previous task) only PRINTS the tip; this task adds the opt-in that performs the install, and must confirm the target agent is supported (fall back to printed instructions otherwise).

Note: `picopilot-audio` describes the v2 audio workflow but the audio commands themselves are v2 — the skill can ship with the overview now and be filled as audio lands, OR scope its content to what exists; builder's call, noted.

## Acceptance criteria

- [ ] The `picopilot-*` skills are generated by command group (incur `sync: { depth: 1 }`) and carry each group's workflow discipline (esp. the art loop + the non-multimodal fallback, and the `#include` discipline).
- [ ] `picopilot init --install-skills` runs incur `skills add`; the target-agent support is confirmed, with a printed-instructions fallback for unsupported agents.
- [ ] **Shared-write isolation (REQUIRED — this is the one shared write):** `--install-skills` tests REDIRECT the skills-install target to a TEMP dir (drive via incur `cli.serve(argv, {env})` with an overridden HOME/skills env, and/or incur `--no-global` to a temp project path) AND assert the REAL agent skill dirs (`~/.claude` etc.) are BYTE-UNTOUCHED after the run. Name the lever; the install runs IN-PROCESS so the test process's own env governs the target.
- [ ] Default `init` (no flag) still writes nothing to shared dirs (regression: the isolation obligation is scoped to `--install-skills`).
- [ ] Tests cover skill generation + the isolated `--install-skills` install, mirroring the repo's test style.
