## Problem Statement

A developer wants to build PICO-8 games with an AI agent (Claude Code, Codex, etc.). Today the agent is good at PICO-8 Lua logic but fails in three predictable ways, all rooted in the same cause — the agent cannot perceive or safely manipulate the cart's binary sections, and gets no feedback loop:

1. **Token bloat.** The agent writes verbose Lua and silently blows past the 8,192-token budget; it has no cheap way to see *where* the cost is.
2. **Blind art.** Sprites/gfx live as hex blobs. The agent "draws" without ever seeing the result, so pixel art comes out broken, and it cannot self-correct.
3. **Blind audio.** SFX/music are hex blobs too; the agent cannot compose in a form it reasons about, nor hear the result.

There is no single, agent-installable tool that turns every cart section into LLM-friendly text AND gives the agent the verify/render-back loop it needs to self-correct.

## Solution

`picopilot` — a single TypeScript CLI (built on the `incur` framework) that is simultaneously a CLI for humans, an MCP server, and a set of auto-installable agent skills (incur provides all three from one definition). Its job is to be the **transpile-and-verify layer** between the agent's strength (text) and PICO-8's reality (binary cart sections), plus the glue that gives the agent eyes (gfx render-back), ears (audio render), and an acceptance gate (`verify`).

From the user's perspective: they point their agent at a cart folder, run `picopilot init`, and the agent now has skills + commands to scaffold, write low-token Lua, edit sprites as a readable character grid (and *see* them via a rendered PNG), compose audio as text, run the cart headless, and gate every iteration on a single `verify`. Heavy lifting for token counting / linting / cart parsing / PNG conversion is delegated to the `shrinko8` tool (shell-out to user-installed Python `shrinko`, a load-bearing dependency for the code-quality/conversion commands); the shrinko-free commands (scaffold, gfx grid/render, run, audio transpile) work without it, and the shrinko-backed ones fail with a structured, actionable result rather than crashing (the two-tier model is documented, not hidden).

## User Stories

1. As a developer, I want to run `picopilot init` in an empty folder and get a ready-to-edit cart: a `main.lua` wired via `#include` (so the agent edits plain Lua, never the binary `.p8`), a generated `AGENTS.md`/`CLAUDE.md` carrying the PICO-8 API reference, and a `picopilot.json` config, so the agent starts with the highest-leverage context already in place. `init` does NOT run `git init` (or initialise any VCS): version control is orthogonal to scaffolding a cart, the user may use jj/hg/no-VCS or be scaffolding inside an existing repo (where a nested `git init` is harmful), so `init` only PRINTS a one-line tip ("`git init` to add version control") — instruct, not mutate, the same posture as skills (US #2).
2. As a developer, I want `picopilot init` to PRINT clear instructions for making my agent discover picopilot's skills/MCP (the manual / symlink path) BY DEFAULT — not silently write into my agent's shared skill dirs — so the tool is a good citizen of `~/.claude` / agent dirs and respects how I manage skills (e.g. manual symlinks). I want an explicit opt-in `picopilot init --install-skills` that runs incur's `skills add` to actually install them, so auto-install is a deliberate choice, not a surprise mutation of a shared location. (This is also more portable: skills/MCP discovery differs across agents — Claude Code / Codex / others — so instructing is more robust than auto-installing.)
3. As an agent, I want `picopilot tokens` to report `{tokens, pct, chars, compressed}` against the 8,192 budget (via shrinko8 `--count`), with a CTA to `minify` when over budget, so I can detect and locate bloat each iteration.
4. As an agent, I want `picopilot lint` to surface PICO-8-specific issues (undefined globals, unused/duplicate locals, missing `_init/_update/_draw`) as structured findings (via shrinko8 `--lint`), so I catch the classic gotchas before running.
5. As an agent, I want `picopilot minify` to run SAFE minification (shrinko8 `--minify-safe-only` BY DEFAULT, never aggressive-minify silently) and report the before/after token delta, so I can reclaim budget without changing behaviour. (This is the only v1 command that MUTATES the user's Lua, so safe-only-by-default and an explicit delta report are required; aggressive minification, if ever offered, must be an explicit opt-in flag.)
6. As an agent, I want `picopilot gfx show <sprite>` to render a sprite from `__gfx__` into a readable character grid (`.` = transparent, `0-F` = the 16 PICO-8 colors), with a CTA to `gfx set`, so I can EDIT it precisely (exact, round-trippable, token-cheap — the SVG-for-pixels source).
6b. As an agent, I want `picopilot gfx render <sprite|sheet>` to emit an UPSCALED (nearest-neighbour to ~256px), true-16-colour-palette PNG to a known path, so a multimodal agent can actually LOOK at the pixels and JUDGE whether the sprite reads as intended — because, exactly as with SVG, coding models GENERATE structured visual source well but READ it back poorly, so the char grid is the edit surface and the rendered PNG is the judge surface. The art loop is `gfx render` (look) → `gfx set` (fix via grid) → `gfx render` (re-look), wired as CTAs. The encoder is a shrinko-FREE TS hex→PNG codec using PICO-8's fixed 16-colour palette, so the eyes-loop stays dependency-light and lives in v1-core. (`gfx render` is for VIEWING; it is distinct from `gfx export`, which is a raw 128x128 spritesheet round-trip for external tools.)
6c. As an agent whose model CANNOT read images, I want the `picopilot-art` skill to instruct me to fall back to reasoning over the `gfx show` grid when I can't view the `gfx render` PNG, so the eyes-loop degrades honestly. (The image-reading capability is the AGENT'S, not the tool's — `picopilot` cannot detect it, so the tool ALWAYS emits both the PNG path and the grid, and the SKILL carries the "view it, else imagine the grid" branch.)
7. As an agent, I want `picopilot gfx set <sprite>` to write a character grid back into the cart's `__gfx__`, so I can fix art in the same text loop I read it in — and for sprites 128-255 (which alias the bottom half of `__map__`), I want a SMART refusal: writes are allowed freely when the overlapping map region is empty/all-zero or when an instruction/config authorises overlap, but REFUSED with a structured `{ok:false, reason:"map-overlap", detail:"sprite 130 aliases __map__ rows 40-41 with N non-zero tiles", remedy:"pass --allow-map-overlap, set allowMapOverlap in config, or use sprites 0-127"}` + nonzero exit when real map tiles would be clobbered and nothing authorised that loss, so I never SILENTLY destroy map data I can't see. (Warn-and-succeed is rejected: an agent treats exit-0 as done and ignores warnings.)
7b. As a developer/agent, I want a per-cart `picopilot` config (e.g. `picopilot.json`) that the agent READS for policy and is FREE to write to match higher-level instructions — including `allowMapOverlap` (a rarely/once-set project setting, e.g. "our map is tiny, let sprites use the overlap") — so configuring the project is not nannied. The smart-refuse is what keeps this freedom safe: it only ever stops the agent in the genuine data-loss corner (real tiles at risk AND no instruction/config authorising it), which is a surface-to-human point, not the empty-overlap or instruction-covered cases.
8. As an agent, I want `picopilot gfx import/export` to convert between the cart's spritesheet and a 128x128 PNG (via shrinko8's `spritesheet` format), so art can round-trip to image tools.
9. As an agent, I want `picopilot sfx from-mml` to transpile compact picopilot-MML text into `__sfx__` and merge it into the cart, and `picopilot music from-patterns` to assemble an ordered list of up-to-4 SFX-channel references into `__music__`, so I can compose audio in a form that can actually EXPRESS PICO-8's per-note waveform/volume/effect (which ABC structurally cannot) instead of hand-writing hex. (ABC is dropped from the authoring path: it is score-level and cannot name waveform/volume/effect, so an ABC-authored SFX is timbre-less — "blind audio in a nicer font.")
10. As an agent, I want `picopilot audio render` to export a WAV, so the developer (or a future eval) can hear the result and I can iterate on feedback.
11. As an agent, I want `picopilot run` to launch PICO-8 headless, capture `printh` debug output and the exit state, with a CTA to `lint`/`tokens` on failure, so I have a closed run-test loop.
12. As an agent, I want a single `picopilot verify` that runs tokens + lint + cart-integrity and returns one structured pass/fail envelope, so I have a stable STATIC acceptance gate to drive each iteration toward. `picopilot verify` is a CLI FEATURE the tool ships for the user's CART (distinct from the agent-runner work-contract gate that gates picopilot's OWN development — same word, different level). It is STATIC BY DESIGN: it does NOT execute the cart, so its envelope MUST self-scope ("static gate: tokens + lint + integrity; passing does NOT mean the cart runs") and a green `picopilot verify` CTAs toward `picopilot run` ("static checks pass — now confirm it boots"), so the agent is led from well-formed to actually-runs rather than treating static-green as terminal. With shrinko absent it returns `gate-incapable` (see US #14b), never green.
13. As a developer, I want every command to emit token-efficient TOON output by default (incur) with `--json`/`--format` available, so agent sessions stay cheap and humans still get readable output.
14. As a developer without shrinko8 installed, I want the shrinko-free commands (`init`, `gfx show/set`, `run`, audio transpile) to work unchanged, and the shrinko-backed commands (`tokens`, `lint`, `minify`, `gfx import/export`) to fail with a STRUCTURED result — `{ok:false, reason:"shrinko-not-found", remedy:"pip install shrinko", needs:["python>=3.8"]}` and a nonzero exit code — rather than crashing or returning a hollow success, so I know exactly which capability is gated and how to enable it.
14b. As an agent running `picopilot verify` on a machine without shrinko8, I want verify to return a distinct `gate-incapable` result (nonzero exit, NEVER green) instead of silently skipping the token/lint checks and reporting pass, so the acceptance gate can never pass hollowly by omitting the #1 failure mode (token bloat) it exists to catch.
15. As a developer without PICO-8 installed, I want all text-only commands (`tokens`, `lint`, `gfx show/set`, transpiles, AND `picopilot verify` — which is static and never runs the cart) to work, and `run`/`audio render`/PNG-or-bin `export` to fail with a STRUCTURED `{ok:false, reason:"pico8-not-found", remedy:"set PICO8_PATH or install PICO-8", needs:["pico8"]}` result + nonzero exit (not a crash, not a hang), so the binary dependency — a licensed, paid binary with no pip/npm install path — is a soft, well-signposted boundary mirroring the shrinko boundary.
16. As an agent, I want picopilot to ship skills (`picopilot-overview`, `picopilot-code`, `picopilot-art`, `picopilot-audio`, `picopilot-debug`) loaded on-demand by command group, so I get the hard-won workflow knowledge (the `#include` discipline, reading the token breakdown, the `gfx render`→look→`gfx set`→fix loop incl. the non-multimodal fallback, composing in picopilot-MML) without bloating every turn.

## Implementation Decisions

- **Distribution + runtime (DECIDED).** Primary install/run path is `npx picopilot ...` (zero-install, always-latest, agent-friendly — an agent invokes it in a cart folder and never hand-types a long name, so the package name length is a non-issue). Alternatives for humans: `npm i -g picopilot` or `bun`. The Python shrinko dependency is INDEPENDENT and pip-installed: the honest two-line story is `npx picopilot` (the tool) + `pip install shrinko` (the code-quality/conversion features). Every `shrinko-not-found` structured result MUST carry the EXACT remedy `pip install shrinko` (note: PyPI package is `shrinko`, module is `shrinko8` — the install command uses `shrinko`), never a vague "install shrinko8".
- **Skills discovery is INSTRUCT-by-default, install-on-opt-in (DECIDED; incur capability VERIFIED).** incur's `skills add` / `mcp add` are built-in: `mcp add` works with Claude Code / Cursor / Amp out of the box and `--agent <name>` targets a specific agent; skill grouping (our US #16 `picopilot-overview/code/art/audio/debug`) is incur's `sync: { depth: 1 }` (one skill file per top-level command group). NOTE: `skills add` defaults to a GLOBAL install (`--no-global` installs to the project) — this is exactly why the default `init` only PRINTS instructions and `init --install-skills` is the opt-in that performs the global write. The task building `--install-skills` must still confirm the target agent is supported and fall back to printed instructions otherwise.
- **Naming (DECIDED, Q5).** npm package `picopilot` (the brief author owns the existing `picopilot@0.0.0` placeholder on npm, so the bare name is available), binary `picopilot`, skill namespace `picopilot-*`. Noted-but-accepted footnote: an unrelated `coder/picopilot` exists (a Simon-Willison-blogged VS Code Copilot demo) — different category, no registry conflict since we own the npm name; a searchability footnote only, not a blocker.
- **One artifact via `incur`.** A single `Cli.create('picopilot', …)` with subcommand groups. incur gives MCP (stdio + HTTP `/mcp`), `skills add`, `--llms` manifest, TOON output, Zod arg/output schemas, CTAs, and token pagination for free. We do NOT build a separate MCP server or separate skill bundles — they fall out of the same definition.
- **Layered engine.** `engine/shrinko` (adapter shelling to shrinko8 for count/lint/p8↔lua/spritesheet PNG/section merge), `engine/gfx` (TS codec: `__gfx__` hex ↔ char grid), `engine/audio` (TS codec: picopilot-MML → `__sfx__` hex + pattern-list → `__music__` hex), `engine/pico8` (headless run + capture + export, needs user's PICO-8).
- **Reuse, don't reimplement.** shrinko8 is consumed as an external CLI (shell out to user-installed `shrinko8` / `python -m shrinko8`), not ported to TS — its tokenizer/minifier/cart-parser track PICO-8 versions and porting would be a perpetually-stale effort.
- **shrinko8 invocation = SHELL-OUT to user-installed Python shrinko (DECIDED, Q1).** Rejected: vendoring the "WASM build" — verified that it is NOT a standalone wasm module but the full Python source running on Pyodide (CPython-on-WASM, ~10MB+, multi-second cold start, owns Pyodide/Pillow version drift); too heavy a tax on a tool whose pitch is a cheap per-iteration loop. Rejected: auto-detect both. Chosen: a single shell-out adapter; `picopilot` declares Python + `pip install shrinko` as a load-bearing dependency for the code-quality/conversion commands, and is HONEST about it (one doc line) rather than pretending zero-install.
- **Two-tier capability, per-command structured failure (B1, DECIDED, Q1).** shrinko-FREE commands (`init`, `gfx show/set`, `run`, audio transpile) NEVER touch shrinko and always work. shrinko-BACKED commands (`tokens`, `lint`, `minify`, `gfx import/export`) detect shrinko AT THEIR OWN SEAM; absent, each returns a structured `{ok:false, reason:"shrinko-not-found", remedy, needs}` envelope + nonzero exit — not a crash, not a stub that reports success. Rejected B2 (a front-door gate that refuses everything without shrinko): it would couple shrinko-free commands to a dependency they don't use (more code, not less) and bake in a precondition that a later loosening would break; the clean "one requirement" mental model B2 wanted is recovered with a single README/skill line, not by gating working commands.
- **`verify` with shrinko absent = `gate-incapable`, never green (DECIDED, Q1).** Since `verify` runs tokens + lint (both shrinko-backed), an absent shrinko means it CANNOT check the #1 failure mode. It returns a distinct `gate-incapable` result (nonzero exit), categorically separate from `pass` and from `fail` — it must never report green by skipping its most important checks. This makes shrinko effectively required for `verify` specifically, while `gfx`/`run`/`init` stay genuinely shrinko-free.
- **PICO-8 boundary = OPTIONAL in v1, structured failure when absent (DECIDED, Q3).** PICO-8 is a licensed paid binary with no pip/npm path; only `run` / `audio render` / PNG-or-bin `export` hard-require it (+ `pico8.dat` for some exports). Absent, those return `{ok:false, reason:"pico8-not-found", remedy, needs:["pico8"]}` + nonzero exit — same structured shape as shrinko-absent, never a crash/hang. All text-only commands incl. `picopilot verify` work without PICO-8.
- **`picopilot verify` is STATIC by design (DECIDED, Q3).** It runs tokens + lint + cart-integrity and NEVER executes the cart ("verify does not imply run"). Risk neutralised by HONESTY, not by adding a run: the envelope self-scopes (states it is static and that passing ≠ the cart runs) and green-verify CTAs to `picopilot run`, so an agent is not misled by the connotation of the word into treating static-green as terminal. NOTE the name collision: `picopilot verify` (this CLI feature, gates the USER'S cart) is NOT the agent-runner work-contract gate (which gates picopilot's own dev) — always written `picopilot verify` to disambiguate.
- **"Agent gets eyes" = char grid (edit surface) + rendered PNG (judge surface), DECIDED.** The char grid (`gfx show`/`gfx set`) is the exact, round-trippable EDIT surface; it is NOT sufficient as the JUDGE surface (an LLM reads serialized hex poorly for gestalt/colour, the same way coding models generate SVG well but read it back poorly — author-observed). `gfx render` emits an upscaled, palette-accurate PNG so a multimodal agent can LOOK and judge. Shrinko-free TS hex→PNG encoder (PICO-8's fixed 16-colour palette), in v1-core. The tool always emits both PNG-path and grid; the `picopilot-art` skill carries the non-multimodal fallback ("view the PNG, else reason over the grid") because image-reading is the agent's capability, not the tool's.
- **CTAs encode the loops.** `gfx render` → `gfx set` → `gfx render` (look-fix-relook); `gfx show` → `gfx set`; `tokens` (over budget) → `minify`; `run` (fail) → `lint`/`tokens`; `picopilot verify` (green) → `picopilot run`. The render-then-fix, detect-then-fix, and well-formed-then-actually-runs loops are expressed as incur CTAs so the agent chains without extra prompting.
- **Char-grid is the gfx "source".** The SVG-for-pixels representation is the per-pixel hex grid (maps 1:1 to `__gfx__`); a later phase may add a semantic-region format (pixelsrc-style).
- **incur natively provides the structured-failure envelope, the config file, and the test isolation lever (VERIFIED).** Our structured failures (`shrinko-not-found`, `pico8-not-found`, `map-overlap`, `gate-incapable`) map directly onto incur's `error({ code, message, retryable, cta })` envelope — not custom plumbing. The per-cart `picopilot.json` (US #7b, `allowMapOverlap`) is incur's built-in `config: { files: [...] }` mechanism (JSON; precedence `argv > config > zod defaults`). Test isolation uses incur's first-class `cli.serve(argv, { stdout, exit, env })` DI overrides to drive commands with an overridden `env` and captured `stdout`/`exit` without touching the real environment.
- **gfx/map overlap = SMART refuse + `--allow-map-overlap` + config provenance (DECIDED, Q4).** Sprites 0-127 (`0x0000-0x0fff`) are gfx-only; sprites 128-255 (`0x1000-0x1fff`) alias `__map__` rows 32-63. `gfx set` on 128-255 INSPECTS the overlapping `__map__` region: empty/all-zero → allow (note the aliasing in the result); non-zero tiles AND not authorised → REFUSE with a structured `map-overlap` result + nonzero exit (the agent STOPS — not a warn-and-succeed, which an agent ignores). Authorisation comes from `--allow-map-overlap` on the invocation OR `allowMapOverlap` in the per-cart `picopilot` config. The config is a rarely/once-set PROJECT setting the agent is FREE to read and write to match higher-level instructions; the smart-refuse is the safety, so freedom is fine — the ONLY guaranteed invariant is "picopilot never SILENTLY overwrites existing map tiles that nothing authorised." Blanket-refuse-all-128-255 rejected (annoys safe carts); warn-and-succeed rejected (agents treat exit-0 as done).
- **Audio = MML-only authored notation + a documented picopilot-MML subset (DECIDED, Q2).** MML is the single PRIMARY and ONLY authored target; ABC is dropped from the authoring path entirely (it cannot express PICO-8's per-note waveform 0-7 / volume / effect 0-7). `mml2abc`-chaining is rejected: chaining through ABC would discard exactly the waveform/effect information MML carries. The target is NOT "some existing MML dialect" (there are many, none a 1:1 fit) but a SMALL, DOCUMENTED picopilot-MML subset tuned to PICO-8's exact capabilities: 8 waveforms, 8 effects, 4 channels, the SFX speed/note-length encoding. `__music__` is assembled structurally (an ordered list of up-to-4 SFX references per pattern), not from a notation.
- **The MML→SFX mapping is DE-RISKED WITH A DESIGN-ARTIFACT SPIKE BEFORE v2 feature work (DECIDED, Q2).** The only true greenfield/algorithmic component (everything else is incur + shrinko glue) gets a spike whose OUTPUT is durable design, not a throwaway proof: (1) a `work/notes/findings/<slug>.md` capturing PICO-8's `__sfx__`/`__music__` binary memory layout as external/domain ground truth (with a `source:` pointing at the PICO-8 wiki memory map + a real decoded cart), and (2) the picopilot-MML subset spec (grammar + the exact MML-construct→waveform/effect/channel mapping table, and the defined behaviour for MML features PICO-8 cannot represent). The v2 audio feature tasks carry `blockedBy: [the spike task]`, turning the transpiler from open-ended risk into "implement this documented mapping." If the spike finds the mapping nastier than assumed, the brief is reshaped BEFORE the v2 command surface is committed.

## Testing Decisions

- Test the TS codecs (`gfx`, `audio`) at the seam: round-trip `__gfx__` hex → grid → hex is identity; `gfx render` produces a deterministic upscaled PNG with the correct PICO-8 palette RGB for known hex input (assert pixel bytes, not just "a file exists"); known picopilot-MML inputs produce known `__sfx__` hex (asserted against the spike's documented mapping table, including the defined behaviour for un-representable MML features); known pattern lists produce known `__music__` hex.
- Test the shrinko8 adapter against BOTH presence and absence (mock/stub the binary): present → parses real output; absent → returns the structured `{ok:false, reason:"shrinko-not-found", …}` result with a nonzero exit, NOT a crash and NOT a hollow success. Never let a test depend on shrinko8 actually being installed on the runner. Additionally assert the shrinko-FREE commands (`init`, `gfx show/set`, `run`, transpiles) succeed with shrinko absent (they must not acquire the dependency).
- Test `verify` with shrinko absent returns the distinct `gate-incapable` outcome (nonzero exit) — NOT `pass` and NOT a silent skip. A test that lets verify report green without shrinko is the gate-as-theatre regression this contract exists to prevent.
- Test the gfx/map smart-refuse at the seam: `gfx set` on a 128-255 sprite whose overlapping `__map__` region is all-zero → succeeds; on one with non-zero tiles and no authorisation → returns the structured `map-overlap` refusal + nonzero exit and leaves `__map__` UNCHANGED (assert the bytes are untouched); with `--allow-map-overlap` or `allowMapOverlap` config → succeeds. The "refusal leaves map bytes untouched" assertion is the silent-corruption regression guard.
- Test the PICO-8 runner adapter with PICO-8 absent (the common CI case): it must return the clear "requires PICO-8" result, not hang or crash. Live PICO-8 runs are a manual/opt-in test tier.
- **Shared-location discipline (work-contract "Task quality rule" — NAME the mechanism, assert the real one untouched):**
  - Default `init` writes ONLY to the cart folder (a temp dir in tests) — it does NOT run `git init` or any VCS command, so there is NO git-config child-process isolation concern for `init`. Assert no writes outside the temp cart dir.
  - `init --install-skills` is the ONLY path that writes to the shared agent skill dirs — and incur's `skills add` defaults to a GLOBAL install (VERIFIED), so this is a real global write. Its test MUST redirect that target to a temp dir (e.g. drive via `cli.serve(argv, { env })` with an overridden HOME/skills env, and/or incur's `--no-global` to a temp project path) AND assert the real agent skill dirs are BYTE-UNTOUCHED after the run. Because this path is OPT-IN, the common default-`init` test does not carry this obligation — the dangerous shared write is scoped to one explicitly-flagged command.
  - Note WHERE each path resolves: the shrinko/PICO-8 shell-outs run in a CHILD process (override the child's env, e.g. `PICO8_PATH` / `PATH`); incur `skills add` runs IN-PROCESS (use incur's `cli.serve(argv, { stdout, exit, env })` DI overrides — the test process's own env/config is what governs the install target). The distinction determines whether overriding a child's env suffices or the test process's own env must be set.

## Out of Scope

- Pixel-art / SFX **generative models** — picopilot gives the agent text representations + a loop, not a trained model (the earlier conversation concluded fine-tuning is not worth it).
- Semantic-region art format (pixelsrc-style) and richer screen-fit static analysis — deferred to v3 (incubating in `work/notes/ideas/` if pursued).
- Publishing to the Lexaloffle BBS — out of scope; picopilot stops at a runnable/exportable cart.

## Further Notes

Phasing is TASKING-ORDER, not a hard module boundary — the tasker decides the actual cut (and MAY spin v2/v3 into their own `briefAfter: [picopilot]` briefs; this brief stays one self-contained document and does not pre-fragment). v1 is almost entirely incur + shrinko8 glue and covers the two biggest failure modes (token bloat + blind art):

- **v1-core (the minimal thesis-proving vertical — task and validate FIRST):** `init`, `tokens`, `gfx show`, `gfx set`, `verify`, skills. This is the smallest slice that gives an agent an on-ramp + token-bloat detection + the see-and-fix art loop + a gate + skill discovery. It is ONE indivisible value proposition (each piece is inert without the others), so it is one phase, not several. `verify` in this slice = tokens + integrity (lint folds in when lint lands).
- **v1-rest (refinements + the PICO-8-gated run — task after core is proven):** `lint`, `minify` (safe-only-by-default, see US #5), `gfx import/export` (PNG round-trip, shrinko-backed), `run` (hard-requires the paid PICO-8 binary, so least exercisable in CI). These refine loops the core already closes, or sit behind the paid binary — so if anything slips, the refinements slip, not the core.
- **v2 (NATURAL future-brief candidate):** audio — FIRST the design-artifact spike (PICO-8 sfx/music memory-layout finding + picopilot-MML subset spec), THEN the features that block on it (`sfx from-mml`, `music from-patterns`, `audio render`). The greenfield/risky phase; if spun into its own brief it should get its own grilling pass.
- **v3 (NATURAL future-brief candidate):** semantic-region art, richer screen-fit checks.
